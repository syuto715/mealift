## Reactive vs proactive offline

Phase 1.6 ships **reactive offline detection** ONLY on the chat
surface — `chatStore.isOffline` flips when an EF call returns
`network_error` (NOT `aborted` — user-initiated cancellation is
not a connectivity event; round 1 fix in Phase 1.2 closed that
race). The chat `[id].tsx` send button gates on `isOffline`; the
banner clears either via the dismiss icon or via the next
successful send.

Advice / routine generation / diagnostic surfaces currently do
NOT participate in `chatStore.isOffline` — they rely on the
underlying EF call returning an AIError (`network_error`,
`gemini_error`, etc.) which their per-surface error UI then
renders (e.g. `AdviceCard` retry button, `RoutineGenerationCard`
error state, diagnostic result error block). The "offline →
disabled send" semantic is chat-specific in v1.5.

**Proactive detection** (`@react-native-community/netinfo`
listener that updates state on SSID change / radio drop, plus a
shared offline gate across all four surfaces) was scoped out in
Phase 1.6 because adding a native module mid-implementation
requires a Syuto-approved native-deps sign-off + Expo prebuild
refresh. Tracked as a v1.5+ polish candidate; matrix entry
`Offline proactive detection → DEFER`.
