# v1.5 Stage 2 — External Restaurant Food DB Epic

Placeholder / stub. Detailed design lands at Stage 2 kickoff,
after Stage 1's AI chat epic has reached Phase 1.4 or so (we
want the chat surface's restaurant-menu lookup behavior in
production before we know the right shape for the restaurant DB).

## Goal (one paragraph)

Self-built hybrid external-restaurant nutrition database that
covers the top Japanese chains (gyudon, izakaya, fast food, café,
convenience stores). Source priority: each chain's own
"栄養成分一覧" publication, augmented by 八訂 ingredients for
gaps. AI estimates are fallback only and explicitly tagged.

## Working assumptions (locked in by v1.5 Stage 1 sign-off,
refined at Stage 2 kickoff)

- **Time budget**: 44-64 hours of implementation work; ships in
  the same v1.5.0 release window as Stage 1.
- **Schema changes**: a single Supabase migration adds the
  external-source columns to `foods` (see §`foods` extensions
  below). No new top-level table — the `foods` row is the SSoT
  for both generic and chain-attributed entries.
- **Takedown ready**: every chain-attributed row carries enough
  metadata (`restaurant_chain`, `source_url`, `takedown_flag`)
  to support a one-query bulk suppression if a chain ever
  objects to redistribution.

## `foods` schema extensions (Stage 2 owns)

```sql
ALTER TABLE foods ADD COLUMN restaurant_chain TEXT NULL;
ALTER TABLE foods ADD COLUMN source_url TEXT NULL;
ALTER TABLE foods ADD COLUMN source_type TEXT NOT NULL DEFAULT 'manual'
  CHECK (source_type IN ('hattei', 'official_chain', 'user_generated', 'ai_estimate'));
ALTER TABLE foods ADD COLUMN takedown_flag INTEGER NOT NULL DEFAULT 0;
CREATE INDEX IF NOT EXISTS idx_foods_chain ON foods(restaurant_chain) WHERE restaurant_chain IS NOT NULL;
CREATE INDEX IF NOT EXISTS idx_foods_takedown ON foods(takedown_flag) WHERE takedown_flag = 1;
```

Drafting checklist 95 (takedown-ready) drives the
`takedown_flag` column + the conditional index for fast
suppression sweeps.

## Source ingest pipeline (sketch)

Per chain:

1. Manual scrape / fetch of the chain's official nutrition page.
2. CSV / JSON transform into the seed format.
3. `INSERT OR IGNORE` at app startup OR a server-side
   `public_foods` admin import (decision pending at Stage 2
   kickoff).
4. Aliases populated from `alias-dictionary.json` extension.

## v1.5 Stage 1 connector

### I6 carry-over (Phase 1.0.2 update): Stage 1 では food DB lookup を撤回、 Stage 2 で正式 introduce

Phase 1.0.2 で Stage 1 §8.2 を rewrite し、 **Stage 1 では food DB
lookup 機能を撤回** した (NewI3 resolution: Option 🅰)。 理由は
`src/infra/repositories/foodRepository.ts → getDatabase() →
expo-sqlite` の import chain が RN/Expo runtime 専用で、 Deno EF
runtime 内では import 不可能だったため。 Stage 1 は Gemini の
公知メニュー知識 + 注釈 tag (「公表値」 / 「[推定値]」 / 「DB
未収録」) のみで運用し、 server-side foods lookup は完全に Stage 2
に委譲する。

### Stage 2 の役割 (Stage 1 から委譲された範囲)

Stage 2 が新規 design 対象として引き受ける項目:

1. **Server-side foods table** (or `server_foods` 新規) を Supabase
   に構築 + 八訂 seed migration (script で 1 回限りの ingestion)。
   client 側 SQLite seed (`src/infra/database/seed/data/foods-mext.json`)
   と同等の row 集合を server 側に複製する。
2. **`coach-chat` / `coach-routine` EF が server-side foods table を
   直接 query** (cross-EF HTTP なし、 in-EF Postgres query)。 Stage 1
   §8.2 で禁止した EF-to-EF HTTP は Stage 2 でも継承 (I6 永続化)。
3. **Prompt instruction を 「[公表値]」 grounding に切り替え**、
   現在の Gemini 公知知識依存から DB-grounded answer に upgrade、
   回答品質を向上。
4. **v1.5+ shared server-side helper module candidate** を本格 design
   する (Stage 1 では前提が無く mention 撤回した)。`estimate-nutrition`
   と coach EFs の両方が import 可能な、 Deno-runnable な共有 helper
   として設計。

### Boundaries inherited from Stage 1

- **No cross-EF HTTP** (I6 永続化): Stage 2 でも EF-to-EF の HTTP
  call は禁止。 Helper module 共有 OR in-EF Postgres query のみ。
- **Cross-runtime resource availability verification** (Drafting 99):
  Stage 2 が helper module を新規作成する際、 RN/Expo + Deno EF の
  両 runtime で動作可能か dependency graph を事前に verify する
  (Phase 1.0.1 → 1.0.2 で踏んだ落とし穴の再発防止)。
