## Retrofit decisions deferred to post-device-verify

### Retrofit #1 — production `search.tsx` (530 lines)

| Path | Notes |
|---|---|
| (P1) Rewrite `search.tsx` in place to use `searchUnified` | Highest regression risk; not recommended unless `search-v2` UX falls short. |
| (P2) Promote `search-v2` as the tab entry, deprecate `search.tsx` | Recommended. The v2 path strictly supersedes the legacy LIKE-rank flow (cross-source, fuzzy kana, sort, filter, favorites, infinite scroll, badge). |
| (P3) Coexist | Splits UX; carries long-term maintenance cost. Only viable as a transition step inside P2. |

Recommendation: **(P2)** once device verification confirms the v2 path.

### Retrofit #2 — `addFood` backend / `meal_log_items.food_id`

Sprint 2.3.3 surfaced a concern: restaurant menu items aren't in the
`foods` table at runtime, so `meal_log_items.food_id` couldn't reference
them. Sprint 2.4.1 dissolved the concern by adopting the **(gamma)
snapshot-only** path — `food_id = null` is already supported in the v1
schema, and nutrition is embedded directly into the meal-log row. No
backend change is required. The concern is **closed, not deferred**.
