### 2.2 Why bundle Stage 2 into v1.5.0 (vs. ship Stage 1 first)

The chat-side / Syuto decision (DEC-1, §11) is to bundle Stage 2
into the v1.5.0 ship rather than ship Stage 1 first and follow
with v1.6 for Stage 2. The rationale, briefly:

- **User-facing coherence**: Stage 1 + Stage 2 together solve a
  single user pain point — "I went to マクド for lunch, what
  should I record?" Stage 1 alone tells the user how (ミー先生
  estimates); Stage 2 makes it accurate (公式値 lookup). Splitting
  them ships incomplete value at v1.5.0 and forces a follow-up
  release to land the missing half.
- **Stage 2 unblocks the chat / advice quality ceiling**: every
  ミー先生 response that touches an outside-home meal currently
  has to handle the "no DB" case with an `[推定値]` tag (Stage 1
  §8.2). Once Stage 2 lands, the same answers get to cite
  `[公表値]` from the chain's own nutrition disclosure. This is
  qualitative-leap territory for the chat surface.
- **Migration ordering**: the v34 SQLite slot was already
  reserved in the Stage 1 epic as a deferred candidate. Folding
  Stage 2 into the same release means the v33 → v34/35/36
  forward ladder is contiguous instead of "v33 ships in v1.5.0,
  v34/35/36 in v1.6.0 with a months-long gap."
- **Risk**: a larger release surfaces more regressions. The
  bundled QA dogfood doc (Phase 2.7 rewrite) needs to cover
  both Stage 1 + Stage 2 in one pass; the matrix grows from 8
  → 12-15 scenarios. The bundle path is justified ONLY because
  Stage 1 has already been hardened through 37 Codex rounds —
  Stage 2 lands on a stable base, not a moving one.

The defer order (in case scope pressure forces a cut):
- 1st cut: Phase 2.6 (ミー先生 chat lookup tool) — Stage 2 still
  ships standalone search + quick log + AI estimate without
  the chat surface integration; Phase 2.6 becomes v1.6.
- 2nd cut: Phase 2.5 (AI estimation fallback + v1.5+ G LLMClient
  back-port) — Stage 2 ships with a hard DB-miss UX ("not in
  our database yet"); v1.5+ G back-port and v1.5+ B atomic
  counter both fall out of v1.5.0.
- Non-cuttable: Phase 2.1 / 2.2 / 2.3 / 2.4 / 2.7. The
  foundation, seed, search, quick-log, and ship-docs rewrite
  are the minimum viable Stage 2.

## 3. Design A — 5 surfaces

| # | Surface | Purpose | DB read pattern |
|---|---|---|---|
| ① | 外食 + コンビニメニュー search | Fuzzy search across restaurants × menu items × chain prefixes; surfaces in food picker | `restaurant-menu-lookup` EF, server-authoritative |
| ② | Menu detail screen | PFC + 八訂 cross-reference + barcode (コンビニ PB) + 「ミー先生に相談」 entry | Local SQLite mirror (v35) — read-cache populated by ① |
| ③ | Quick log from menu | 1-tap で `meal_log_items` INSERT with `restaurant_menu_item_id` soft FK; existing nutrition log と統合 | Existing `nutritionRepository` write + v36 soft FK column |
| ④ | Restaurant + コンビニ browse | Chain ごとの menu / SKU list、 restaurant_type filter, top-frequent SKUs surfaced first | Local SQLite mirror (v34 + v35) — populated by initial sync |
| ⑤ | ミー先生 chat 内 lookup | チャットで店舗質問 → `lookup_restaurant_menu` tool call → DB return → 記録 + アドバイス | Tool extension on Phase 1.2 `coach-chat` EF; chat stream stays NDJSON |

### 3.1 Sequence sketches

#### Surface ① — menu search

```
[UI: food picker / search input]
  → user types "マクド" or "セブン"
  → existing foods search runs in parallel
  → Stage 2 lookup runs against SQLite mirror first:
       SELECT restaurant_menu_items_local.*
         FROM restaurant_menu_items_local m
         JOIN restaurants_local r ON r.id = m.restaurant_id
        WHERE r.name LIKE ? OR r.aliases LIKE ?
              OR m.name LIKE ?
        ORDER BY m.use_count DESC
        LIMIT 30
  → if mirror returns 0 rows AND online → fire
    `restaurant-menu-lookup` EF for cross-mirror reconciliation
  → if mirror is stale (e.g. brand-new chain) → EF response
    upserts both mirror tables before returning
  → results surface alongside foods / public_foods / barcode_foods
    in the existing result list, tagged with a `🍴` icon + chain
    name prefix
```

#### Surface ⑤ — ミー先生 chat lookup

```
[chat]
  user: "今日のお昼マクドのビッグマック食べた"
  → ミー先生 model sees this turn + system prompt that now
    declares the `lookup_restaurant_menu` tool
  → model emits a tool_call event in the NDJSON stream:
     { event: 'tool_call', name: 'lookup_restaurant_menu',
       args: { restaurant_name: 'マクド', menu_name: 'ビッグマック' } }
  → EF intercepts, calls `restaurant-menu-lookup` server-side
    (same EF surface ① uses), gets PFC payload
  → EF emits a tool_result event:
     { event: 'tool_result', name: 'lookup_restaurant_menu',
       result: { chain_id, menu_id, pfc, calories, ... } }
  → model resumes generation with the lookup result in
    context, producing an answer with `[公表値]` tag
  → user confirms "記録する" → quick-log path (surface ③)
    fires INSERT
```

The tool call is **synchronous server-side** (EF intercepts +
re-calls Gemini with the tool_result in context). Latency budget
is 2-3 s end-to-end — the lookup is a local Postgres query, not
a downstream EF. NDJSON streaming protocol (Stage 1 Drafting 96)
adds the `tool_call` + `tool_result` event types — taking the
total vocabulary to 6 events (see §7.3); the client renders an
"ミー先生がメニューを確認中..." spinner between the two new
events.

### 3.2 Offline behavior

- **Read paths** (① + ② + ④): SQLite v34 + v35 mirrors are
  populated on first launch via a one-shot
  `restaurant-menu-lookup` EF call with `mode: 'initial_sync'`.
  Subsequent reads hit the mirror first. Mirror staleness check
  fires every 7 days (when online) to pull deltas. **No** chain
  + menu DB read requires a live connection after the initial
  sync.
- **Write paths** (③): quick-log goes through the existing
  `nutritionRepository.addMealLogItem` which already has the
  offline-safe `sync_queue` integration. The new
  `restaurant_menu_item_id` soft FK column gets enqueued the
  same way. Apply transition (chat / advice / generation
  parallel) is online-required for the Stage 1 surfaces; Stage
  2 quick-log can fully apply offline.
- **AI estimate fallback** (Phase 2.5): if the DB lookup returns
  no row AND online, the EF can fall back to `lookup_restaurant_menu`
  on Gemini with the chain + dish name as context. Offline DB
  miss returns the same hard "not in our database yet" UX as
  the Stage 1 `[データベース未収録]` tag.

## 4. Architectural foundation

### 4.1 Data acquisition pipeline (DEC-2, §11)

**Hybrid AI-scrape + Syuto spot check + bulk import**, run once
per chain by Claude Code:

1. **Scrape** — Claude Code's `web_fetch` against each chain's
   official nutrition disclosure page. Output is a JSON list
   of `{menuName, calories, protein, fat, carb, fiber?, sodium?,
   ...}` rows.
2. **Spot check** — Syuto manually validates 10-20% of rows
   against the same source page. Mismatches > 5% require a
   re-scrape (likely a parsing error, not a source error).
3. **Bulk import** — validated rows commit to
   `restaurant_menu_items` via the seed migration
   `20260520000003_restaurant_menu_seed.sql`. The migration is
   idempotent (`ON CONFLICT DO UPDATE`) so re-running it is
   safe.
4. **Ongoing maintenance** — chain nutrition pages change at a
   low rate (quarterly menu refreshes typical). A v1.6+
   `restaurant_menu_item_versions` view shows diffs vs. the
   live `restaurant_menu_items` so Syuto can confirm what
   changed before promoting the new seed.

The pipeline is fundamentally human-in-the-loop. Claude Code is
the bulk extractor; Syuto is the QA gate. Neither is a
substitute for the other. Drafting 113 candidate (§12) crystallizes
this lesson if Phase 2.2 confirms the workflow.

### 4.2 Restaurant + コンビニ seed data ingest scripts (Phase 2.2a + 2.2b)

`scripts/seed/` is the Phase 2.2a landing site for the pipeline
(Codex round 1 Phase 2.2a Nit — path was `scripts/seed-restaurant-menu/`
in the original draft, normalized to `scripts/seed/` at landing
time since the kickoff prompt directed the simpler path):

1. `scripts/seed/restaurant-menu-scraper.ts` reads the chain's
   official nutrition disclosure page via the pluggable fetcher,
   parses (HTML / JSON dispatch), validates PFC band + Atwater
   consistency, and writes one JSON record per chain to
   `scripts/seed/data/${chain_slug}.json`. Failures write a
   markdown post-mortem to `scripts/seed/data/_failed/${chain_slug}.md`.
2. `scripts/seed/spot-check-helper.ts` selects a deterministic
   10-20% sample (seeded by chain_slug) and emits a markdown
   review sheet at `scripts/seed/spot-check/${chain_slug}-sample.md`.
3. `scripts/seed/build-seed-migration.ts` aggregates every
   validated `data/*.json` into the single SQL migration at
   `supabase/migrations/20260520000003_restaurant_menu_seed.sql`
   using deterministic v5 UUIDs (SHA-1 + namespace) so the same
   chain_slug always yields the same id across re-builds. Every
   INSERT uses `ON CONFLICT DO UPDATE`; menu items also bump
   `version` so the Phase 2.1 audit-capture trigger snapshots
   the prior payload into `restaurant_menu_item_versions`.

The 36 entities + (estimated) 30-50 menu items each = ~1,200-
1,800 rows total. SQL file size estimate: ~200-300 KB. Inline
into the migration repo (not S3 / external storage) — small
enough to stay version-controlled.

### 4.3 AI estimation fallback (Phase 2.5, v1.5+ G application)

The Phase 1.1 `LLMClient` abstraction (`src/infra/llm/`) is
the natural home for `restaurant-menu-estimate`. Phase 2.5 ships
this EF + a parallel `LLMClient.estimateRestaurantMenu()`
interface method.

This is also the natural Phase 2.X moment to back-port the
existing 5 Gemini EFs (v1.5+ G in Stage 1's §13). The 5 EFs
(`estimate-nutrition`, `estimate-nutrition-vision`,
`nutrition-advice`, `generate-workout-menu`,
`generate-weekly-report`) move from their bare-fetch pattern
to `LLMClient.generateXyz()`. The back-port is gradual:
Phase 2.5 lands `restaurant-menu-estimate` + the first 2
back-ports (most-trafficked: `estimate-nutrition` +
`nutrition-advice`); the remaining 3 back-port in Phase 2.7
polish window. Stage 2 +1.5+ G crystallization documented in
§11 DEC-5.

### 4.4 八訂 cross-reference (existing foods DB → restaurant menu)

`restaurant_menu_items` rows store PFC at the menu-level
(complete dish). The 八訂 (Standard Tables of Food Composition
in Japan, 八訂) breakdown is **dish-type-dependent**:

- **`dining`** restaurants (FF / 牛丼 / 寿司 / ファミレス /
  カフェ / その他): the menu dish ingredient decomposition is
  not officially published. The user-facing surface shows the
  公式 PFC value; the optional `ingredient_decomposition_json`
  column carries a best-effort estimate (computed via Gemini
  in Phase 2.5 OR null if no estimate exists yet). The
  decomposition is a v1.6+ enrichment goal, not a v1.5
  requirement.
- **`convenience`** SKUs (セブン / ファミマ / ローソン etc.):
  the back-of-package nutrition label is the source of truth.
  PFC is direct. `ingredient_decomposition_json` is generally
  null (SKUs are processed foods; ingredient list is on the
  package but ingredient grams aren't disclosed).
- **`cafe_bakery`** (スタバ / タリーズ / ドトール / コメダ /
  プロント / フレッシュネス): mixed model. Bakery items track
  `dining`; coffee + drinks track `convenience`.

The `restaurant_type` enum (§5.3) drives the per-type rendering
strategy. The 八訂 cross-reference is **read-only** from Stage
2's perspective — Stage 2 doesn't mutate the existing `foods`
table.

### 4.5 User-contributed defer pattern (v1.6+ design intent)

The existing `public_foods` table has a full RLS submission
lifecycle (Phase 1 of v1.4 ship): pending_review → approved
/ rejected, moderation by service-role admin. A future
`public_restaurant_menu_items` table would mirror that lifecycle
but is **deferred to v1.6+** to keep Stage 2 within the v1.5.0
ship window. The 36-entity Tier 1 + future Tier 2 + Tier 3
(another 30+ chains each) cover the long tail; user submissions
are mostly for niche regional chains or new SKUs not yet in
Tier 1.

### 4.6 Open Food Facts barcode reconciliation (コンビニ PB)

Stage 1 v1.4 baseline ships an Open Food Facts barcode cache
(`barcode_foods` SQLite table + `barcodeFoodRepository`,
ODbL-attributed). Convenience store private-brand (PB) products
(e.g. セブンプレミアム のホットドッグ) sit at the intersection
— they ARE in OFF, AND they ARE in the Stage 2 seed.

The reconciliation policy: when a `barcode` column exists on a
`restaurant_menu_items` row AND the same barcode is in
`barcode_foods`, the picker UX prefers the Stage 2 row
(chain-branded, official nutrition disclosure) and folds the
OFF row out of the search results to avoid duplicate listings.

**Codex round 1 Important #6 fix** — mirror annotation alone
is insufficient. The client search pipeline merges results from
multiple sources (`foods` / `public_foods` / `barcode_foods` /
`restaurant_menu_items_local`) and the duplicate can still
surface in the merged result list even if the Stage 2 mirror
"knows" about the preference. The actual reconciliation must
happen at the merge step: after collecting candidates from all
sources, group by `barcode`, and when a barcode appears in BOTH
`restaurant_menu_items_local` AND `barcode_foods`, drop the
`barcode_foods` row from the merged list. Phase 2.3 search UX
implementation owns this client-side dedupe.

```sql
create table public.restaurants (

-- Phase 2.1 — set_updated_at trigger so delta pulls (§3.2)
-- can rely on `updated_at` advancing on every write. Codex round
-- 1 Important #4 fix.
create or replace function public.set_updated_at_restaurants()
returns trigger language plpgsql as $$
begin
  new.updated_at := now();
  return new;
end $$;
create trigger set_updated_at_restaurants_trg
  before update on public.restaurants
  for each row execute function public.set_updated_at_restaurants();

create index restaurants_type_idx on public.restaurants (restaurant_type);
create index restaurants_aliases_idx on public.restaurants using gin (aliases);

-- Public read; service-role write only.
alter table public.restaurants enable row level security;
create policy "Public read non-takedown restaurants"
  on public.restaurants for select
  using (takedown_flag = false);
```

#### `restaurant_menu_items`

```sql
create table public.restaurant_menu_items (
  id uuid primary key default gen_random_uuid(),
  restaurant_id uuid not null references public.restaurants(id)
    on delete cascade,
  name text not null,
  aliases text[] not null default '{}',  -- ['ビッグマック', 'BigMac']
  category text,  -- 'バーガー' / 'サイド' / 'ドリンク' (chain's own taxonomy)

  -- PFC, mirrors existing foods schema
  serving_size_g real not null default 100,
  serving_unit text not null default 'g',
  serving_description text,  -- '1 個' / '1 杯' / '1 食' etc.
  calories_per_serving real not null check (calories_per_serving >= 0 and calories_per_serving < 3000),
  protein_g real not null default 0 check (protein_g >= 0 and protein_g < 200),
  fat_g real not null default 0 check (fat_g >= 0 and fat_g < 300),
  carb_g real not null default 0 check (carb_g >= 0 and carb_g < 500),

  -- Optional micronutrients (mirrors public_foods extended set)
  fiber_g real,
  sugar_g real,
  salt_g real,
  sodium_mg real,
  saturated_fat_g real,
  cholesterol_mg real,

  -- Convenience PB SKUs carry a barcode; dining menu items don't.
  barcode text,

  -- Optional ingredient decomposition for dining items;
  -- Phase 2.5 may fill this via Gemini estimate. Null = unknown.
  ingredient_decomposition_json jsonb,

  -- Source attribution; identifies where the row's data came from.
  source text not null check (source in (
    'official_disclosure', 'package_label', 'ai_estimate', 'manual'
  )),
  source_url text,
  source_captured_at timestamptz,

  -- Version tracking (chain menu refreshes).
  version int not null default 1,

  -- Search ranking signal.
  use_count int not null default 0,

  takedown_flag boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  unique (restaurant_id, name)
);

create index restaurant_menu_items_restaurant_idx
  on public.restaurant_menu_items (restaurant_id, use_count desc);
create index restaurant_menu_items_aliases_idx
  on public.restaurant_menu_items using gin (aliases);
create index restaurant_menu_items_barcode_idx
  on public.restaurant_menu_items (barcode)
  where barcode is not null;

-- Phase 2.1 trigger — see set_updated_at_restaurants above.
create or replace function public.set_updated_at_restaurant_menu_items()
returns trigger language plpgsql as $$
begin
  new.updated_at := now();
  return new;
end $$;
create trigger set_updated_at_restaurant_menu_items_trg
  before update on public.restaurant_menu_items
  for each row execute function public.set_updated_at_restaurant_menu_items();

alter table public.restaurant_menu_items enable row level security;

-- Codex round 1 Critical #3 fix — chain-level takedown must
-- suppress the chain's menu items too. The policy joins back
-- to `restaurants.takedown_flag = false` so flipping the chain
-- flag immediately drops all child menu rows from public read.
create policy "Public read non-takedown menu items"
  on public.restaurant_menu_items for select
  using (
    takedown_flag = false
    and exists (
      select 1 from public.restaurants r
       where r.id = restaurant_menu_items.restaurant_id
         and r.takedown_flag = false
    )
  );
```

#### `ai_lookup_logs` (Codex round 2 OPEN #1 fix — full column set)

Phase 2.1 migration `20260520000002_create_ai_lookup_logs.sql`.
User-scoped quota + idempotency replay store, analogous to the
existing `ai_usage_logs` table from v1.4 but tailored to the
Stage 2 lookup path (carries the cached response payload so the
idempotency replay returns the SAME items without re-running
the DB query).

```sql
create table public.ai_lookup_logs (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  function_name text not null,            -- 'restaurant-menu-lookup' /
                                          -- 'restaurant-menu-estimate'
  input jsonb not null,                   -- { chainName, menuName,
                                          --   barcode, mode } parsed
                                          --   request body
  response_payload jsonb,                 -- cached items[] returned
                                          --   on idempotency replay;
                                          --   nullable for failed
                                          --   requests
  response_status int not null,           -- HTTP status code; quota
                                          --   gate filters on 200
  idempotency_key text,                   -- nullable so retention
                                          --   cleanup can NULL it
                                          --   without losing the
                                          --   quota row
  created_at timestamptz not null default now()
);
```

### 5.2 SQLite mirrors

**SQLite array storage**: Codex round 1 Important #3 fix —
`text[]` is a Postgres-only type; SQLite has no native array.
The Supabase-side `aliases text[]` column maps to the local
side via a **side table** pattern:

- `restaurants_local` carries the chain row WITHOUT the aliases
  array. A separate `restaurant_aliases_local` table holds one
  row per alias (`restaurant_id`, `alias`, `alias_lower`) with
  an index on `alias_lower` for prefix-match lookups.
- `restaurant_menu_items_local` likewise mirrors the menu row
  without aliases; `restaurant_menu_item_aliases_local` carries
  the per-menu aliases.

The sync helper (Phase 2.1) splits the Supabase `aliases` array
into individual side-table INSERTs at upsert time. Client-side
search queries (§3.1) join the side tables:

```sql
select m.*
  from restaurant_menu_items_local m
  join restaurants_local r on r.id = m.restaurant_id
  left join restaurant_aliases_local ra
    on ra.restaurant_id = r.id
   and ra.alias_lower like ?
  left join restaurant_menu_item_aliases_local ma
    on ma.menu_item_id = m.id
   and ma.alias_lower like ?
 where (r.name like ? or ra.alias_lower is not null
        or m.name like ? or ma.alias_lower is not null)
 order by m.use_count desc
 limit 30;
```

v34 — `restaurants_local` + `restaurant_chain_categories_local`
+ `restaurant_aliases_local`: read-cache populated by initial
sync. Same shape as Supabase tables minus the `takedown_flag`
column (the sync helper filters it out at fetch time, plus
joins the chain takedown gate per §6 RLS). Phase 2.1 lands
these.

v35 — `restaurant_menu_items_local` +
`restaurant_menu_item_aliases_local`: read-cache; ditto.
`use_count` is bumped client-side at quick-log time (Phase 2.3
read-back path) so the search rankings stay responsive even
offline, but the server-side count is the authoritative one
(client `use_count` resets to server's value on next sync).

v36 — Codex round 1 Critical #2 fix — extends BOTH layers:
- SQLite-side `meal_log_items` (the existing v1 table — NOT
  `meal_log_items_local`; Codex round 1 Nit #3 fix removed
  the stale `_local` suffix) gains a
  `restaurant_menu_item_id TEXT NULL` soft FK column.
- Supabase-side `user_meal_log_items` (the server mirror that
  `sync_queue` pushes to — see
  `20260505000000_create_user_sync_tables.sql`) gains the same

  `20260520000004_alter_user_meal_log_items.sql`. Without this

Existing rows (pre-Stage-2) are unaffected (NULL by default).
The column drives the "this entry was logged from
セブンイレブン's ハーゲンダッツ バニラ" UX badge on the meal

## 7. API design (Edge Functions)

### 7.1 `restaurant-menu-lookup` EF (Phase 2.3)

12-step pattern lifted from Phase 1.5's `coach-routine`. Codex
round 1 Critical #1 fix — `ai_lookup_logs` is **now created in
Phase 2.1** (migration `20260520000002`) so this EF has a real
quota log to read / write from day one. The atomic counter RPC
(Phase 2.5, v1.5+ B) is a perf upgrade on top of the existing
table, not a prerequisite. Stage 1 idempotency contract
preserved — Codex round 1 Important #2 fix:
**`Idempotency-Key` is a REQUIRED HTTP header** (matches
`coach-chat` / `coach-advice` / `coach-routine`); the body
does NOT carry an `idempotencyKey` field.

1. Auth (Bearer token verify; immutable).
2. **Idempotency replay** — read `Idempotency-Key` from
   request headers (required; reject with 400 if missing).
   Lookup `ai_lookup_logs` rows where `user_id = ?` AND
   `idempotency_key = ?` (Phase 2.1 migration carries a
   partial unique index on `(user_id, idempotency_key)
   WHERE idempotency_key IS NOT NULL` — Drafting 109).
   On hit, return the cached `response_payload` directly
   (the value was persisted by the original call's STEP 8;
   see Drafting 98 — replay path must NOT re-run the
   downstream DB query, otherwise the contract becomes "same
   request, different result" if the underlying menu rows
   change between the two calls).
3. Plan gate (`restaurantMenuLookup` boolean, default true for
   all tiers; the monthly cap differs per tier — §9).
4. UTC-monthly quota gate (matches `coach-chat` /
   `coach-advice` / `coach-routine` parity); counts
   `ai_lookup_logs` rows where `user_id = ?` AND
   `response_status = 200` AND `created_at >= utcMonthStartISO`.
5. Parse body: `{ chainName?, menuName?, barcode?, mode? }`.
   `mode` is `search` (fuzzy) or `initial_sync` (delta pull).
6. Build SQL query: alias gin index search OR exact barcode
   lookup OR full delta pull.
7. Execute; return ranked rows (use_count desc within fuzzy
   match score).
8. INSERT `ai_lookup_logs` row carrying
   `{ idempotency_key, function_name: 'restaurant-menu-lookup',
   input: {...}, response_payload: { items, mode,
   cachedUntil }, response_status: 200 }` — quota counted HERE
   (Drafting 98). The `response_payload` write IS load-bearing
   for STEP 2 replay (Codex round 3 OPEN #2 fix — the column
   exists in §5.1 DDL but STEP 8 has to populate it so STEP 2
   has something to return on the next idempotency hit).
9. — (no Gemini call for surface ①; lookup is direct DB read).
10. — (no schema validation needed; rows come from trusted
    DB).
11. — (no UPDATE needed; lookup is read-only).
12. Return `{ items, mode, cachedUntil }` to client.

Compensation: STEP 3 / 4 gate failures return 403 / 429 with
the standard error envelope. STEP 6-7 SQL failures return 500.
No mutable state to roll back; the lookup is read-only.

### 7.2 `restaurant-menu-estimate` EF (Phase 2.5)

Used when the DB lookup misses. Mirrors Phase 1.5

### 7.3 ミー先生 chat tool extension (Phase 2.6)

Extends `coach-chat` (Phase 1.2) without renaming or
restructuring. The system prompt gets a new section:

```
【使用可能なツール】
- lookup_restaurant_menu(restaurant_name, menu_name) — 外食 +
  コンビニメニューの 公式 PFC + 八訂値を返す。 ユーザーが
  「マクド」 「セブン」 「ビッグマック」 等と言及した時に
  自動で呼び出す。
```

The Gemini model emits the tool call via the
`tools` + `function_declarations` API surface (already supported
by `gemini-2.5-flash`). The EF intercepts the tool_call payload,
calls `restaurant-menu-lookup` server-side (NOT through the
client), and feeds the result back into the model's context for
a second-turn generation. The NDJSON stream now carries **6
event types** (Codex round 1 Nit #1 fix — count corrected from
the earlier "5-event" wording):

1. `meta` — assistantMessageId, conversationId, model.
2. `chunk` — text delta.
3. `tool_call` — name + args (transient; for UI spinner).
4. `tool_result` — name + result (transient; resumes streaming
   after).
5. `done` — inputTokens / outputTokens / finishReason.
6. `error` — code + message + recoverable.

Drafting 96 (streaming wire protocol locked at design SSoT)
applies — the NDJSON vocabulary lands in Phase 2.6 BEFORE
implementation kickoff. Drafting 108 (cross-EF audit) applies
— the tool call is server-internal (EF-to-EF), so no client-
side EF chaining; this matches the §8.2 I6 fix pattern from
Stage 1.

**Client diff inventory (Codex round 1 Important #1 fix)** —
the 6-event vocabulary expansion lands in these client-side
files at the same time as the EF:

| Layer | File | Change |

## 14. ミー先生 integration spec (Phase 2.6)

Phase 2.6 ships the `lookup_restaurant_menu` tool call extension
to `coach-chat`. Spec lockdown for the implementation kickoff:

### Tool declaration (in `coach-chat` system prompt)

```
【ツール: lookup_restaurant_menu】
引数:
  - restaurant_name (string, 必須): 「マクド」 「セブン」 等
  - menu_name (string, optional): 「ビッグマック」 等。 未指定の
    場合、 上位 N 件を返す
返り値:
  {
    matches: [
      { chain_name, menu_name, calories, protein_g, fat_g,
        carb_g, serving_description, source, source_url }
    ],
    miss: boolean  // true = DB に無し、 推定値を出力すべき
  }
使用ガイドライン:
  - ユーザーが外食 / コンビニのメニュー名を述べたら自動で呼ぶ。
  - 結果を引用する際は 「[公表値]」 タグを付ける。
  - miss=true の場合のみ AI 推定 (「[推定値]」 タグ) で答える。
```

### Wire protocol (NDJSON, extension of Stage 1 §3 vocabulary)

```
event: tool_call
  → { event: 'tool_call', call_id: uuid, name: 'lookup_restaurant_menu',
      args: { restaurant_name: 'マクド', menu_name: 'ビッグマック' } }
event: tool_result
  → { event: 'tool_result', call_id: uuid, name: 'lookup_restaurant_menu',
      result: { matches: [...], miss: false } }
```

The `call_id` chains the request/response pair across the stream.
The EF intercepts the tool_call line, calls the DB lookup, and
emits the tool_result line; the model then resumes with a
second turn.

### Latency budget

- Tool call → DB lookup → tool result emission: **< 800 ms**
  (local Postgres + 1 indexed query).
- Total user-perceived added latency vs Stage 1 chat: **< 2 s**
  (tool call + second Gemini turn).

### UI

`MessageBubble` (Phase 1.2) gets a transient state between
tool_call and tool_result: render a 「ミー先生がメニューを
確認中...」 spinner inside the assistant bubble while the
tool resolves.

### Pro-only gating

`miCoachRestaurantTool: false` for Free + Plus, `true` for Pro
(DEC-5, §9). Free + Plus users get the standard ミー先生 chat
without the tool call; they have to manually use the
restaurant search surface (① / ②). The Pro-only positioning
is the Stage 2 daily-advice analogue.

## 15. 法的 / 倫理 considerations

### Attribution

Every `restaurant_menu_items` row carries a `source` enum +
`source_url`. The seed migration (Phase 2.2) cites each chain's
公式 nutrition disclosure URL + 取得日 in the
`attribution`/`attribution_url` columns on the `restaurants`
table. UI footers credit chains with "公式情報を元に作成" + the
chain name + retrieval date when surfacing menu details.

### Commercial restaurant 名の使用 (Fair use)

- Chain names (商標) are used in a **nominative fair use** sense
  ("これは マクドナルド のメニューです") which does NOT require
  trademark holder permission under Japanese law.
- The app does NOT imply endorsement, sponsorship, or affiliation
  with any chain. The 「公式情報を元に作成」 phrasing reinforces
  this.
- Logos are NOT used. Only the chain name as plain text.

### データ精度の disclaimer

A footer on the menu detail screen reads:
> 公式情報を元に作成しています。 メニューの変更や調理時の
> ばらつきにより実際の値と異なる場合があります。

Phase 1.6 a11y polish + medical disclaimer pattern carries
forward — the disclaimer is exposed to VoiceOver with a
`accessibilityRole="text"`.

### Takedown procedure

If a chain requests removal (rare; nominative fair use is
defensible but the chain may still object), the `takedown_flag`
column on `restaurants` flips to `true` via service-role admin.
The public-read RLS policy immediately suppresses all rows for
that chain. Phase 2.7 ship checklist includes a "verify takedown
SQL works" step.

### Privacy policy update

The v1.5 ship checklist (Phase 1.7) already flagged that the
Stage 1 chat / diagnostic / routine generation / advice
surfaces expand the PII boundary sent to Gemini. Stage 2 does
NOT add new PII categories — the `restaurant-menu-lookup` EF
takes only `chainName` / `menuName` / `barcode` / `mode` in
the request body (per §7.1 STEP 5; Codex round 1 Nit #4 fix
— the earlier wording incorrectly claimed nutrition context
was sent, which conflated lookup with the Phase 2.5
`restaurant-menu-estimate` EF that DOES forward `UserContext`
to Gemini for the fallback path). The privacy policy update
required for Stage 1 covers Stage 2.

### User-contributed content moderation (deferred)

When v1.6+ unblocks user submissions (§4.5, §13 A), the
moderation lifecycle from the existing `public_foods` system
re-applies. Stage 2 v1.5.0 doesn't ship user submissions, so
moderation isn't a v1.5.0 launch concern.
