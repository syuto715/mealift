## Promotion sequence (Syuto-san)

Execute in order. Each step is reversible up to the App Store
submit.

1. **Push the v1.5 commits to origin**
   ```
   git push origin main
   ```
   Phase 1.0 through 1.7 split into the commit count agreed at
   the chat-side sign-off (typically 7-10 commits).

2. **Cut the release-candidate tag**
   ```
   git tag -a v1.5.0-rc1 -m "v1.5.0 release candidate 1 — Stage 1 (ミー先生)"
   git push origin v1.5.0-rc1
   ```

3. **EAS production build**
   ```
   eas build --platform ios --profile production
   ```
   `appVersionSource: "remote"` + `autoIncrement: true` in
   `eas.json` handle the buildNumber automatically (v1.4
   pattern preserved). No manual buildNumber bump in
   `app.config.ts`.

4. **TestFlight self-dogfood (24-48 h)**
   Install the EAS build via TestFlight. Re-run the v1.5 QA
   scenarios end-to-end on the production build, not the dev
   build — Stage 1 has more prod-only configuration (Edge
   Function URLs, RevenueCat product ids, vault secrets) than
   any prior release.

5. **App Store Connect — v1.5.0 record**
   - Bundle ID: `com.mealift.app` (unchanged)
   - ASC App ID: 6762738456 (unchanged from v1.4 ship; verify
     in ASC)

     Japanese-only system prompt + Japanese-only persona; no
     i18n in v1.5)

## Rollback path

If a critical issue surfaces in TestFlight:

- The local `main` is the single source of truth for the
  implementation side. Revert the offending commit, re-tag,
  re-build. Do NOT amend or rewrite history that has already
  been pushed.
- If the App Store submission has already gone in, use the ASC
  "Reject this binary" flow before review picks it up.

If an issue surfaces in the Edge Function path post-launch:

- The four Stage 1 EFs deploy independently — a hotfix to
  one (e.g. tightening the schema validator in coach-routine)
  can be `supabase functions deploy coach-routine` without
  touching the client build.
- The migration path is forward-only; emergency rollbacks
  require a new ALTER migration, not a rollback statement.
