# v1.5 Roadmap

Carry-over items from the v1.4 ship plus the v1.5 candidates surfaced
during the v1.4 implementation. Each entry is a "what / why-deferred /
rough scope" stub — concrete recon passes precede implementation as
in v1.4.

This document is the single source of truth for "v1.4 said next-time"
references scattered across code comments + commit messages. When a
v1.5 item ships, mark it `Done — see <commit>` here and drop the
TODO from the source code.

## Architectural deferred items (confirmed v1.4 defer)

### Auth + telemetry

- **Sentry integration** — auth + EF error visibility in production.
  v1.4 shipped the local 3-tier visibility pattern (user-facing alert
  + `__DEV__` console + Sentry TODO marker) but the production
  telemetry side is still a stub. Scope: install + auth tier 2/3
  observation events + the Codex-flagged `partial_success: true` tag
  on barcode retries.
- **Codebase-wide try/catch audit** — the generic-error masking anti-
  pattern caught in `barcode.tsx` and `auth listener` came from
  recon-first scans. v1.4 fixed those sites; a third pass would sweep
  the remaining services / repositories. Pattern 11 facet 6 (Error
  visibility 3-tier) application boundary.
- **Auth Tier 2/3 production monitoring** — the v1.4 Tier 2 probe +
  Tier 3 startAutoRefresh reorder need real production telemetry to
  confirm the spurious-SIGNED_OUT recovery rate. Hook the listener's
  recover / commit branches into the Sentry tag above.

### Routes SSoT Phase 3

- **Bulk inline → ROUTES migration** — v1.4 Stage 5.1 crystallized
  the `ROUTES` SSoT (`src/constants/routes.ts`) but only migrated the
  9 callsites that Stage 5.1A touched plus the 5 high-frequency Pro
  CTA components. The remaining ~45 inline `/(tabs)/...` /
  `/(onboarding)/...` references across feature screens are queued
  for v1.5 once the routing surface (Auth Tier 2/3, RNTL coverage,
  deep-link plumbing) settles.

### Test infrastructure

- **RNTL component tests** — v1.4 chose Path C (jest 30 + manual
  config + pure-helper extraction) because jest-expo's published
  versions (54.0.2 through the current 55.x latest + 56.x next) all
  pin internal deps at `@jest/globals@^29` / `babel-jest@^29` /
  `jest-environment-jsdom@^29` / `jest-snapshot@^29`. When expo
  upstream catches up to jest 30 the calculus changes — at that
  point, adopt jest-expo + add full `<RootLayout />` render tests for
  the auth bootstrap + listener wiring.
- **jest-expo migration trigger** — monitor jest-expo's package.json
  for the jest 30 bump. When it lands, audit the existing 1993-test
  suite under the preset for behavior parity.

### Vision / AI enhancement

- **TOCTOU atomic counter RPC** — both `estimate-nutrition` and
  `estimate-nutrition-vision` use a count-then-insert quota gate that
  is not atomic under concurrent requests. Stage 4 Turn 2 Codex
  surfaced this; the fix is a CTE-style RPC that performs the gate +
  insert in one transaction. Apply to both EFs.
- **Vision multi-prompt design** — the v1.4 Vision EF returns only
  dish name + serving description + ingredient grams; the user fills
  PFC manually. A second pass could chain Vision output → text
  estimate-nutrition input to auto-fill PFC, but the design needs to
  resolve quota counting (one user action → two EF calls) and the
  UX flicker (Vision modal → text estimate modal → final modal).
- **`UNIT_CATEGORIES` filter by category** — v1.4 Stage 4 Phase 4F
  crystallized the 7-option unit segments but did not wire the
  category filter (food.serving_unit → which segments to show). The
  picker shows all 7 today; a follow-up could narrow per food type.

### Subscription / monetization

- **Android RevenueCat parity** — v1.4 ships iOS-only as the
  monetization surface. Android requires a separate Google Play
  Billing integration through RevenueCat + the offering / package
  IDs reflected in the Android entitlement check. Estimated 5-8h.
- **Vision-specific entitlement split** — Stage 4 reused the existing

  When the free tier opens up a "monthly trial" path on Vision (a
  design candidate, not committed), split into `aiTextEstimate` +
  `aiVisionEstimate` so the two quotas can diverge cleanly.

## Pattern crystallization candidates

The v1.4 implementation surfaced these patterns; v1.5 is the
natural moment to crystallize each into the Handbook:

- **Pattern 18 facet 11** (preset adoption version compat verify) —
  Stage 5.2 Path C is the canonical example. Before adopting a
  preset that bundles a parallel jest / babel / etc. version, verify
  the embedded version's compatibility with the host project's
  installed version. The example: `jest-expo` pinned at `@jest/
  globals@^29` while the host runs jest 30; the right call was to
  decline the preset and crystallize the relevant config manually.
- **Pattern 11 facet 7-ish** — Failure-domain isolation inside the
  try / catch. Stage 5.3 Codex pass caught a `try` that wrapped
  both a probe and a downstream RC call; the RC failure was bleeding
  into the probe-failure recovery branch. Narrow the try to the
  single failure surface it intends to observe.
- **4-actor SSoT culture** — chat-side Claude design + Agent recon +
  Codex pass review + Syuto-san sign-off. v1.4 saw four Critical
  catches across Stage 4 / 5.2 / 5.3 from the Codex pass alone —
  every one of them would have shipped a regression without the
  pair. Worth codifying the cadence as a Handbook section.
- **Drafting checklist accumulation** — items in the 80-90 range
  during v1.4 are getting hard to navigate as a flat list.
  Reorganize into a matrix (failure-mode × layer) before adding
  more in v1.5.

## Stage-4-specific Vision carry-overs

The v1.4 Vision EF deliberately ships a minimal scaffold:

- Vision identifies the dish + estimates a 1-serving weight.
- PFC values are filled in by the user via the manual tab.

The full "Vision → PFC auto-fill" flow needs:

1. A multi-prompt design (Vision recognizes dish, then text
   estimate-nutrition fleshes out PFC from the identified name).
2. UX reconciliation (one camera tap → one modal, not three).
3. A quota model that charges the right number of calls per user
   action.

Track this as a v1.5 design item rather than an implementation one
until the UX direction is settled.

## Architectural humility lesson (v1.4 retrospective)

The v1.4 implementation cycle reconfirmed that the four-actor SSoT
culture compounds rather than overlaps. Across the stage-by-stage
Codex passes:

- **Stage 4 Critical 1**: `scan-dish` was routing back to the home
  screen instead of the meal-context-preserved add screen. The
  Vision result would sit in the store and get consumed by an
  unrelated meal log on the next `add.tsx` focus. Pre-dogfood catch.
- **Stage 5.2 Critical 1**: the extracted `bootstrapAuthSession`
  catch path was demoting a persisted-local user on a transient
  network blip — the pre-extract code only did that when not already
  authenticated. Logic-preservation invariant violated. Pre-dogfood
  catch.
- **Stage 5.3 Critical 1**: the factory listener's
  `isLocalOnly()=true` branch was calling `setUnauthenticated()`
  when the pre-5.3 inline listener explicitly did nothing in that
  case. Local-only users would have been silently demoted. Pre-
  dogfood catch.
- **Stage 5.3 Critical 2**: the `INITIAL_SESSION` callback fires
  immediately on subscribe; the factory listener was treating its
  null session as a logout even when bootstrap had just preserved
  the persisted local auth. A persisted-local user would have seen
  a "logged out" flash on every cold start. Pre-dogfood catch.

Every one of these was the kind of defect that arrives only after
a user reports the symptom. The pair caught all four before
TestFlight, on the same day the implementation landed. Document
the cadence; preserve the discipline.
