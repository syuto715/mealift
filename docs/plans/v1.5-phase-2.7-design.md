# v1.5 Phase 2.7 — Design (Sprint 2.7.1) 🎯

Phase 2.7 = v1.5 ship runway. Kickoff anticipated a 5–8-sprint program
centered on **LLMClient unification + back-port + ship docs**.
Reconnaissance on 2026-05-23 produced two opposite findings:

1. **LLMClient back-port is a no-op.** The abstraction the kickoff
   wanted to build already exists, was already architecturally
   deliberate, and the remaining wrapper functions are *intentional*
   richer-envelope extensions, not bare-fetch oversights.
2. **Drafting 172 fan-out is the real work.** Phase 2.6 hardened only
   `coach-chat`. The four sister EFs (`coach-advice`, `coach-routine`,
   `nutrition-advice`, `estimate-nutrition-vision`, plus likely
   `generate-weekly-report`) still ship with no L3 defense paragraph,
   no L4 input cap, no L5 secret scrub.

So Phase 2.7 collapses on the LLMClient axis but *expands* on the
security axis — and the security work is precisely what `security.ts`
extracted in Sprint 2.6.2 was designed to enable.

## Reconnaissance findings

### LLMClient abstraction status

| Asset | Location | Status |
|---|---|---|
| `LLMClient` interface | `src/infra/llm/types.ts:196-224` | ✅ **DEFINED** — three methods (`chat`, `generateStructured`, `generateAdvice`) |
| `GeminiFlashClient` implementation | `src/infra/llm/geminiFlashClient.ts:566 lines` | ✅ **IMPLEMENTS** — class signature line 31: `export class GeminiFlashClient implements LLMClient` |
| Used by `chatStore` | `src/stores/chatStore.ts` | ✅ via `new GeminiFlashClient()` |

The kickoff's premise — "5 bare-fetch files need a unified abstraction"
— is incorrect. The unification already exists, was the explicit
architectural goal of Phase 1.1 (see `docs/plans/v1.5_stage_1_ai_chat
_epic.md §4 LLMClient abstraction`), and shipped to production months
ago.

### Wrapper-function reality

`coachAdviceClient.ts` (line 4-8 docstring):

> Client-side wrapper over the `coach-advice` Edge Function.
> **Distinct from `LLMClient.generateAdvice()`** (which is the abstract
> surface returning just text) — this richer wrapper returns the full
> advice envelope so the store can cache by `(scope, period_start)` and
> the UI can render a generated_at timestamp.

`routineGenerationClient.ts` (line 1-5):

> Client-side wrapper over the `coach-routine` Edge Function.
> Returns the full generation envelope so the store can persist a
> draft + later reference it by `generationId` for apply / discard.

These are not bare-fetch oversights — they are **deliberate, documented
design** that sits *alongside* the LLMClient interface, providing the
richer envelopes that the production stores need for persistence /
caching / status reasoning. Refactoring them through
`LLMClient.generateAdvice` / `generateStructured` would either:

- Break the envelope (regression), or
- Add a redundant pass-through layer (cosmetic refactor with no
  business value)

→ **No-action recommendation for "LLMClient back-port".**

### `contextBuilder` is shared utility

`contextBuilder.ts:260 lines` exports `buildUserContext` +
`buildProfileSnapshot` + four summary builders. It is reused by all
three call paths (`chatStore` via `GeminiFlashClient`, `coachAdviceStore`
via `fetchCoachAdvice`, `routineGenStore` via `fetchRoutineGeneration`)
and never imports any client itself. Already the right design.

### `types.ts` is shared types

271 lines of `ChatMessage` / `ProfileSnapshot` / `UserContext` /
`LLMClient` / etc. — the contract surface every other file imports
from. Already the right design.

## 🛡️ Drafting 172 fan-out — the actual Phase 2.7 work

Phase 2.6 landed L3 SYSTEM_PROMPT defense + L4 length cap + jailbreak
log + L5 secret scrub on `coach-chat`. The helpers were extracted to
`supabase/functions/coach-chat/security.ts` precisely so sister EFs
could pick them up. Audit on 2026-05-23:

| Edge Function | Accepts user text? | L3 defense paragraph | L4 length cap | L5 secret scrub | Status |
|---|---|---|---|---|---|
| `coach-chat` | yes (chat) | ✅ | ✅ | ✅ | Phase 2.6 complete |
| `coach-advice` | yes (context-driven prompt) | ❌ | ❌ | ❌ | **GAP** |
| `coach-routine` | yes (context + goals) | ❌ | ❌ | ❌ | **GAP** |
| `nutrition-advice` | yes | ❌ | ❌ | ❌ | **GAP** |
| `estimate-nutrition-vision` | yes (text-prompt + image) | ❌ | ❌ | ❌ | **GAP** |
| `generate-weekly-report` | partially (context-only) | ❌ | ❌ | ❌ | **GAP** if any user-supplied text reaches the prompt |

Each gap is a fresh attack surface. A jailbreak that fails on
coach-chat may succeed against coach-advice with the same payload
because coach-advice's prompt has no defense paragraph and its EF has
no scrub.

### Fan-out unit of work (per EF)

The Sprint 2.6.2 + 2.6.3 changes are now a template:

1. **Lift** the existing `coach-chat/security.ts` into a shared
   location (`supabase/functions/_shared/llmSecurity.ts`) so every EF
   can import the same validators. Drafting 162 honored — no schema
   touch, just a module move + re-export.
2. **Per EF**, in 1-3 commits each:
   - Append the L3 defense paragraph (with EF-specific tweaks) to the
     EF's `SYSTEM_PROMPT` constant.
   - Wire `checkUserContentLength` + `detectJailbreakHints` after the
     EF's `messages` / `prompt` body parse.
   - Wire `scrubSecrets` at every chunk emit site (streaming EFs) or
     at the final response body (non-streaming EFs like
     `coach-advice` / `coach-routine`).
3. **Per EF**, add a `__tests__/security.test.ts` matching the
   `coach-chat` template.

## `_shared/llmSecurity.ts` lift design (Sprint 2.7.2 first commit)

Current location: `supabase/functions/coach-chat/security.ts`
Target location: `supabase/functions/_shared/llmSecurity.ts`

Approach:

1. Copy file content to the new path (literal copy, no rename of
   exports).
2. Replace `coach-chat/security.ts` with a one-line re-export from the
   new path so the existing `coach-chat/index.ts` import + tests keep
   working without changes:

   ```ts
   export * from '../_shared/llmSecurity.ts';
   ```

3. Each sister EF (Sprint 2.7.2-3) imports from the new shared path
   directly:

   ```ts
   import {
     MAX_USER_CONTENT_CHARS,
     checkUserContentLength,
     detectJailbreakHints,
     scrubSecrets,
   } from '../_shared/llmSecurity.ts';
   ```

4. `coach-chat/__tests__/security.test.ts` and `redTeam.test.ts`
   either keep importing from `../security` (via the re-export) or
   move their import path to the shared module. Either works; choose
   the less-noisy diff.

## `[unnumbered new candidate]` — Cross-EF security policy fan-out pattern

Phase 2.6 isolated the hardening pattern into `security.ts` so it
could be lifted. Phase 2.7 makes the lift explicit and applies it
across the entire EF surface. The pattern: **whenever one EF receives
defense-in-depth treatment (Drafting 172), the same treatment must
fan out to every sibling EF accepting user input within the same
model-class boundary (text / vision / streaming-chat).** Security is
a server-side policy that follows the EF surface, not the client
abstraction — even when the clients share a unified abstraction
(Phase 2.7's case), hardening doesn't propagate automatically.

This is not Drafting 172 itself — it's the *deployment discipline* that
keeps 172 honest across a multi-EF backend. Candidate for chat-side
numbering at Phase 2.7 close.
