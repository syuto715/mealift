# v1.4 Phase G QA Scenarios

Manual dogfood scenarios that must pass on a real device before the
v1.4 build can be promoted from TestFlight to App Store review.
Eight scenarios cover the surface that v1.4 changed (Stage 1-5.3
implementation) plus the regression-critical paths the changes
intersected with.

Each scenario is a "happy path + boundary" pair. Mark each
checkbox as `[x]` when verified on the listed device. Anomalies go
in the "Notes" line at the bottom of the scenario.

## Pre-flight

- TestFlight build installed (latest from EAS).
- Test account A (free / no trial) and account B (Pro) both
  reachable from the build configuration.
- Device clock and time zone left at the device's normal settings.
  Do NOT toggle these mid-scenario — they cross-talk with HealthKit
  / nutrition aggregation.

## Scenario 1 — Home screen states

Free / Trial / Plus / Pro must all render correctly.

- [ ] **Free**: ProTeaser visible at bottom; greeting reflects time
  of day (4-tier); calorie progress bar renders; water input row
  accepts a tap and posts the value (the v1.3.5 narrow-target fix
  must hold — tapping the row should NOT race the TextInput focus).
- [ ] **Trial**: ProTeaser visible; trial days-remaining pill
  reads correctly (TrialBadge); upgrade route lands on
  `/(tabs)/settings/subscription`.
- [ ] **Plus**: ProTeaser hidden; gold accent on plan badge.
- [ ] **Pro**: same as Plus + Pro-only features unlocked.

Boundary: leave the app in background for >5 min, return, check
that the home screen rehydrates without an empty/skeleton state.

## Scenario 3 — Nutrition screen + 3-tab food add

The food-add screen has three top-level tabs (検索 / スキャン / OCR)
with the search tab nesting six sub-tabs.

- [ ] Search tab: all six sub-tabs (検索 / マイ料理 / よく使う /
  お気に入り / 手入力 / テンプレ) navigable. Search returns results;
  manual entry saves; template apply works.
- [ ] Scan tab: Pro user — tapping「AI で料理を撮影」 opens the
  Vision camera (scan-dish route).
- [ ] Scan tab: free / trial user — same CTA presents the Pro
  upsell instead of opening the camera.
- [ ] OCR tab: tapping「ラベルを撮影」 opens the OCR camera
  (scan-label route). A nutrition-label photo populates the
  ServingQuantityModal pre-fill.
- [ ] Barcode scan: scan a known product, add to a meal.
- [ ] **2nd-scan regression check**: leave the barcode screen,
  return, scan again — the camera must still register the read
  (the v1.3.5 useFocusEffect / cameraKey fix must hold).
- [ ] Barcode unknown product registration: fill the form, tap
  「登録して追加」. Then force-fail the meal-log write (no clean
  way to simulate this in production; verify via the partial-
  success Alert copy review).

## Scenario 7 — Vision + OCR meal logging

The Vision + OCR meal-logging flows ship for the first time in
this build. Each path needs a live test with the Edge Function
deployed.

- [ ] **Vision happy path**: Pro user taps「AI で記録」 on a meal
  row → camera opens → photo of a known dish → AI analysis runs
  → router.replace → add screen lands on the manual tab with
  dishName + serving grams pre-filled → user enters nutrient
  values → 「追加」 saves to that specific meal.
- [ ] **Vision Pro gate**: free user sees the upsell instead of
  the camera (defense-in-depth: even if a deep link drops them
  on /scan-dish directly, the route's own gate fires).
- [ ] **OCR happy path**: open OCR tab → 「ラベルを撮影」 → photo
  of a nutrition panel → ServingQuantityModal opens pre-filled
  with calories + PFC; saving adds the entry.
- [ ] **3-tier fallback**: photo a clearly unreadable label (Pro
  user) → OCR returns nothing → Vision auto-retries silently →
  on Vision success the manual-tab pre-fill flow runs; on Vision
  failure a graceful 「読み取れませんでした」 alert offers manual
  entry.
- [ ] **Vision payload ceiling**: photograph a very high-res image
  → EF should respond with `image_too_large` (413) → UI surfaces
  the message; the camera-quality=0.4 capture should normally
  stay well under the 1.3M base64 ceiling.

## Scenario 8 — Auth session persistence

The auth Tier 2 spurious-SIGNED_OUT recovery + Tier 3 startAutoRefresh
reorder ship for the first time in this build. The canonical
dogfood checklist
(docs/plans/auth_session_persistence_fix.md line 129-136) applies
verbatim:

- [ ] **Force-quit → relaunch × 10**: signed-in state persists on
  all 10 launches (no kick back to the login screen).
- [ ] **Background ↔ foreground × 5**: signed-in state persists;
  watch for any "logged out" flash on return.
- [ ] **Airplane mode toggle (ON → OFF) while in app**: the auth
  state holds; no infinite spinner, no premature logout. When
  the network returns, any deferred sync work resumes silently.
- [ ] **Cold-start auth bootstrap**: open the app fresh from
  ASC TestFlight install (clean install simulates worst-case).
  The session restoration from persisted storage works.
- [ ] **Local-only mode**: register an account, sign out, register
  a new account as local-only. Force-quit + relaunch must keep
  the local-only state (Stage 5.3 Codex Critical fix — the
  factory listener must NOT clobber local-only on null-session
  events).

## Device coverage

- **iOS primary**: iPhone 14 Pro (or iPhone 15 / 16 Pro if
  available) — current-gen pointer-accuracy + 6.1" layout.
- **iOS legacy**: iPhone SE (3rd gen, 4.7") — narrow-screen wrap
  fallbacks (training header, food row chips).
- **Android**: deferred to a follow-up build. The release config
  ships iOS as primary; Android dogfood is a separate gating
  checklist when that build branches off.

## VoiceOver / a11y

- [ ] **Home screen header**: greeting + date both readable; the
  cloud-upload pending badge announces the count or "no pending
  submissions" rather than reading raw numerals out of context.
- [ ] **Training header buttons**: each button's accessibilityLabel
  reads its full action (not the icon name).
- [ ] **Food add screen**: SegmentedControl announces the active
  tab; sub-tab switches re-announce.
- [ ] **ServingQuantityModal**: numpad keys + gram-mode toggle
  reachable + announce their state correctly.
- [ ] **ProCard / ProInlineCTA / ProTeaser**: the upgrade button's
  accessibilityRole is `button` and its label includes the price /
  trial hint (not just "Plus へ").

## Sign-off

A passing run of all eight scenarios on iPhone 14 Pro plus the
iPhone SE wrap-fallback checks in Scenarios 2 + 3 is the
implementation-side gate for promoting the build to App Store
review. Document any deviations in this file's notes lines; the
implementation team decides whether each deviation is shipping or
a re-spin.
