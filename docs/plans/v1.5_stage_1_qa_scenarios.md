# v1.5 Stage 1 QA Scenarios

Manual dogfood scenarios that must pass on a real device before
the v1.5 build can be promoted from TestFlight to App Store
review. Eight scenarios cover the four new ミー先生 (AI コーチ)
surfaces (chat ① / diagnostic ② / generation ③ / advice ④)
plus the regression-critical paths these surfaces intersect with.

Format mirrors `docs/plans/v1.4_qa_scenarios.md`. Each scenario
is a "happy path + boundary" pair; check `[x]` when verified on
the listed device, and capture anomalies in the trailing Notes
line. The Phase 1.6 input doc
`docs/qa/v1.5_stage_1_manual_dogfood.md` carries the working
matrix; this file is the canonical Stage 1 ship dogfood spec.

## Pre-flight

- TestFlight build installed (latest from EAS, production
  profile). The dev / preview channel does NOT exercise the
  prod RevenueCat configuration nor the prod Supabase project.
- Test account A: **Free / no trial**.
- Test account B: **Plus (active subscription or active trial)**.
- Test account C: **Pro** — exercised in Scenarios 1 / 3 / 4 / 7
  where Pro tier differs from Plus (unlimited chat / 20-month
  generation quota / Pro-only daily advice flag). Stage 1 does
  NOT ship a daily-advice UI surface (§13 F deferred), so the
  "Pro daily advice" scenario row is a flag-presence check
  only — Pro's `aiCoachAdviceDaily` is `true` in FeatureFlags
  but no surface acts on it yet.
- Device clock + time zone left at the device's normal settings.
  Do NOT toggle these mid-scenario — the advice surface buckets
  in profile-tz at the EF, and a mid-flight tz change masks
  bucket-key bugs as flakiness.

## Scenario 1 — Chat surface ① (Phase 1.2)

### Free user (Account A)

- [ ] Open the bottom-tab 「コーチ」 → conversation list +
  「ミー先生」 + 「AI コーチ」 sub-label renders correctly.
- [ ] Quota row reads `今月の残り: 5 / 5` on first install.
- [ ] Tap `+ 新しい会話` → first send works → token-by-token
  render with `meta` → `chunk` → `done` NDJSON flow.
- [ ] After 5 successful sends → 6th tap is blocked (send button
  greyed) + ProInlineCTA `「ミー先生にもっと相談するには Plus へ →」`.
- [ ] Tap CTA → routes to `/(tabs)/settings/subscription`.

### Plus / Pro user (Account B / C)

- [ ] Quota shows `199 / 200` (Plus) after first send; counter
  hidden on Pro (unlimited).
- [ ] Long-press a conversation row → action sheet shows
  `アーカイブ / 削除 / キャンセル`.
- [ ] Archive: row disappears immediately; check Supabase
  `chat_conversations.archived_at` is non-null.
- [ ] Delete: 2-step confirm (`削除` → `削除する`) before the
  row + messages are removed.
- [ ] Force-kill mid-stream → relaunch → the assistant row
  reflects the server-authoritative status (`partial` if Gemini
  was mid-output; `final` if it already committed; `error`
  if the stream errored before the close).

### Boundary — offline

- [ ] Airplane mode → send → reactive offline banner appears,
  send disabled. Dismiss icon clears banner; next successful
  send re-enables it.

### A11y

- [ ] VoiceOver focuses persona header → reads
  「ミー先生 AI コーチ」.
- [ ] Send button hint varies by reason for disable:
  「ネット接続を確認してください」 (offline) /
  「今月のチャット上限に達しました」 (quota) /
  「メッセージを入力してください」 (empty) /
  「ミー先生にメッセージを送信します」 (enabled).
- [ ] Long-press hint reads
  「長押しでアーカイブ・削除メニューを開きます」.

## Scenario 7 — Cross-account logout reset chain

Phase 1.2 + 1.4 + 1.5 + 1.3 each wired their per-user Zustand
store into `authStore.logout`'s lazy-import reset chain
(Drafting 106). Verify the chain:

1. Login Account B → exercise chat / diagnostic / generation /
   advice; each generates at least one persistent artifact.
2. Logout B → confirm:
   - Coach tab opens with empty conversation list.
   - Diagnostic entry shows locked (since A is free) OR fresh
     wizard state (if next account is Plus).
   - `RoutineGenerationCard` shows no current draft.
   - `AdviceCard` shows locked.
3. Login Account A → same as fresh-install state for A; no
   Account B content surfaces.

The 3 stores wired into the reset chain (coachAdvice +
routineGen + diagnostic) AND the chat store's own
account-switch handling are the 4 surfaces that need this
verification.

- [ ] All four surfaces show fresh state for Account A; nothing
  from Account B leaks.

## Pass criteria

### Pre-TestFlight (eight scenarios must all pass)

- All eight scenarios pass on a Plus-tier iPhone (Account B).
- Scenario 1 + 4 + 5 also pass for Free (Account A) and Pro
  (Account C) where applicable.
- Scenario 8 ops checks all green.

### Pre-App Store submission (additional bar)

- Scenario 6 VoiceOver pass on at least the chat thread + the
  diagnostic wizard.
- No regressions on the v1.4 surfaces (home / training basic
  routine flow / nutrition / progress / settings) — spot-check
  one happy path per surface.
- TestFlight build version reads `1.5.0` + buildNumber matches
  EAS auto-increment.

## Rollback

If a critical issue surfaces on TestFlight:
- Revert the offending commit, re-tag, re-build.
- Do NOT amend pushed history.
- If submitted to App Store but not yet in review, use ASC
  "Reject this binary".

If an issue surfaces post-launch:
- Hotfix via EAS Update for JS/asset bug.
- Native code / config bug → new build → ASC version with
  expedited review request.
