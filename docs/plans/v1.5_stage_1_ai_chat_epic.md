## 1. Goals + scope

The Stage 1 epic ships a single coach persona ("ミー先生") that
surfaces across four UI entry points:

| # | Surface | Purpose | LLM call shape |
|---|---|---|---|
| ① | Conversational chat | Free-form Q&A, multi-turn, follow-ups | Streaming response |
| ② | Diagnostic flow | Structured onboarding-like wizard for new users / re-baselines | One-shot at end |
| ③ | Program generation | Routine builder driven by free-text intent | One-shot, JSON-structured |
| ④ | Weekly / daily coach advice | Embedded text block on routine + nutrition screens | Scheduled, one-shot |

The implementation is split across **eight sub-phases** (1.0
through 1.7); this document is Phase 1.0's output.

Out-of-scope for Stage 1 (carried to v1.5 Stage 2 or beyond):

- External-restaurant menu DB construction (Stage 2 epic).
- Migrating the existing four Gemini Edge Functions
  (`estimate-nutrition`, `estimate-nutrition-vision`,
  `generate-workout-menu`, `nutrition-advice`,
  `generate-weekly-report`) to the new LLM abstraction. The
  abstraction is forward-only for Stage 1; back-migration is a
  v1.5+ candidate (see §4).
- Provider switching at runtime. Stage 1 ships Gemini 2.5 Flash
  with the abstraction in place so a future Claude / OpenAI
  client can be added without touching call sites.

### 2.2 Supabase Edge Functions

`supabase/functions/` directory (six EFs total):

| EF | Purpose | Model | Quota |
|---|---|---|---|
| `estimate-nutrition` | Text dish-name → ingredient decomposition | `gemini-2.5-flash-lite` | per-day (Pro) |
| `estimate-nutrition-vision` | Photo dish image → ingredient decomposition | `gemini-2.5-flash` (multimodal) | 20/day Pro |
| `nutrition-advice` | Free-text nutrition coach advice | `gemini-2.5-flash-lite` | per-day Pro |
| `generate-workout-menu` | Workout program JSON generation | `gemini-2.5-flash-lite` | monthly (free 3 / plus 30 / pro 100) |
| `generate-weekly-report` | AI weekly narrative | Gemini | monthly Plus+ |
| `send-push-notifications` | Cron-driven push outbox | n/a | n/a |

All Gemini EFs follow a uniform pattern (sample:
`generate-workout-menu/index.ts`):

1. Bearer auth: extract token from `Authorization`, call
   `userClient.auth.getUser()`.
2. Plan gate: query `profiles.plan` via the admin client; bail
   with `pro_required` / `plus_required` as appropriate.
3. Quota gate: count successful (`response_status = 200`) rows in
   `ai_usage_logs` keyed on `(user_id, function_name)` over the
   relevant window (daily UTC or monthly UTC). Reject with
   `quota_exceeded` if the cap is met.
4. Validate request body.
5. Call Gemini via `fetch(generativelanguage.googleapis.com/...)`.
6. Parse the response (`responseMimeType: 'application/json'` for
   structured calls; raw text otherwise).
7. `finally`: insert a row in `ai_usage_logs` regardless of
   outcome (so abuse / unauthorized attempts are also recorded).

The race window on (3) → (7) is a known TOCTOU; the canonical
mitigation is a CTE-style atomic counter RPC and is tracked in
`docs/plans/v1.5_roadmap.md`. v1.5 Stage 1's new chat / generation
EFs **must use the same pattern**, including the same TOCTOU
limitation, until the atomic counter ships.

### 2.3 Client-side Edge Function caller

`src/infra/services/aiNutritionService.ts` is the SSoT EF caller
(despite the file name — sibling services like
`aiWorkoutService.ts` and `aiWeeklyReportService.ts` import
`callEdgeFunction` and `AIError` from here):

- `AIError` class: `code: AIErrorCode | message: string | status:
  number | details?: Record<string, unknown>`.
- `AIErrorCode` union: `'unauthorized' | 'invalid_token' |
  'pro_required' | 'plus_required' | 'quota_exceeded' |
  'invalid_request' | 'gemini_error' | 'internal_error' |
  'network_error' | 'not_configured' | 'no_equipment' |
  'validation_failed' | 'aborted'`.
- `callEdgeFunction<TReq, TRes>(path, body, { signal? })`:
  Bearer-tokened POST to `${SUPABASE_URL}/functions/v1/${path}`,
  parses structured error codes from the response body, surfaces
  `AbortError` as `AIError('aborted', ...)`.
- `decomposeFromImage`, `decomposeRecipe`, `fetchNutritionAdvice`,
  `estimateDishNutrition`: thin wrappers + response-shape
  validation.

Streaming-shaped responses do NOT exist today: every existing EF
returns a single JSON object. The Stage 1 streaming chat surface
is the first time this caller layer needs chunked-fetch / NDJSON
streaming support (see §4 for the wire format chosen, and §3 for
the line vocabulary).

### 2.5 Food DB schema + ingestion

`foods` table is local-SQLite-first (v1). Seed data comes from
two pipelines:

- `scripts/import-mext-foods/` — Japanese 八訂 (MEXT 2023
  supplement) Excel workbook → parsed JSON → `INSERT OR IGNORE`
  at app startup. Stable IDs `mext_NNNNN`; aliases populated from
  `alias-dictionary.json`. ~2,500 rows.
- `scripts/food-import/` — USDA / brand seeds, separate ID space
  (`fd_NNN`).

There is also a server-side `public_foods` table (Supabase) for
user-submitted foods, with its own auto-approval pipeline
(`supabase/migrations/.../create_public_foods.sql`,
`auto_approval_trigger.sql`).

The `foods` schema has columns appropriate for individual foods
(`brand`, `barcode`, per-serving macros + extended nutrients).
The `dishes` schema is the "composite" level (recipes / multi-
ingredient meals), with `dish_ingredients` rows holding the
breakdown. Stage 2's external-restaurant DB sits closer to the
`foods` shape than the `dishes` shape; the schema additions for
restaurant lineage / takedown go on `foods` (or a sibling
`foods_external` if it turns out the lineage columns shouldn't
mix with user-customized rows — see Stage 2 design).

Search pipeline: `foodRepository.searchFoods(query, limit)` is
the canonical entry; it returns ranked rows by name match with
`food_aliases` join. `foodRepository.findByExactName` is the
exact-match version used by the AI estimate pipeline. No
embedding / vector search today.

### Sequence sketch — chat send (Pattern 20 + Pattern 11 application, C2 fix)

```
[UI: user taps Send]
  → chatStore.sendMessage(text, idempotencyKey = uuid v4)
  → optimistic local append (TWO pending rows, each with a
     locally-generated clientTempId so the meta event can bind
     to the right one later):
      { role:'user', content:text, status:'pending',
        clientTempId: uuid() }
      { role:'assistant', content:'', status:'pending',
        clientTempId: uuid() }
  → POST /functions/v1/coach-chat
      Headers:
        Authorization: Bearer <token>
        Idempotency-Key: <uuid v4>
      Body:
        { conversationId, messages, contextSnapshot }

  [Server: coach-chat EF]
    1. Auth: Bearer token verify (immutable check; no mutable state
       touched).
    2. Idempotency check: if Idempotency-Key already mapped to an
       assistantMessageId for this user, short-circuit:
       - Return existing chat_messages records + replay stream
         from the persisted server buffer (or short-circuit to the
         persisted final state if the original stream already
         finished).
       - NO plan gate, NO quota gate, NO Gemini call, NO
         ai_usage_logs INSERT (初回 attempt が既に quota を消費
         済、 replay は free)。
       - Mapping eligible for null-out at 24h; cleared by the
         hourly cleanup job (effective retention = 24h + cron
         lag, up to ~1 hour additional; see §5.1 retention prose).
    3. Plan gate: query profiles.plan; bail with plan-required code
       on mismatch.
    4. Quota gate: count ai_usage_logs over the UTC monthly window;
       reject with quota_exceeded if the cap is met.
    5. INSERT chat_messages (role='assistant', content='',
       status='pending', model, idempotency_key) → assistantMessageId.
       Assistant placeholder を先に INSERT することで、 同じ
       Idempotency-Key を持つ concurrent duplicate request が来た
       場合に `chat_messages_idempotency_key_unique` (partial
       unique index) がここで発火し、 second request を 409 で
       reject。 user INSERT が走る前に止まるので、 user row が
       重複することは無い (race-safe ordering; Phase 1.1 Codex
       round 1 C2 fix で確立)。
    6. INSERT chat_messages (role='user', content, status='final').
    7. INSERT ai_usage_logs (quota counted HERE, before the Gemini
       call). Note: a partial stream still counts against quota —
       the same usage-charged model as the existing four Gemini EFs.
    8. Stream meta line: { event:'meta', assistantMessageId,
       conversationId, model }.
    9. Stream Gemini; accumulate into a server-side buffer; emit
       chunk lines: { event:'chunk', delta: "..." }.
    10. On success:
        - UPDATE chat_messages SET content=<buffer>, status='final',
          output_tokens=N WHERE id=assistantMessageId.
        - Stream done line.
    11. On mid-stream error:
        - UPDATE chat_messages SET content=<buffer-so-far>,
          status='error' WHERE id=assistantMessageId.
        - Stream error line; close connection.
    12. On client disconnect (connection closed by client before
        done/error): 30 s safety timer → if Gemini still streaming,
        abort + UPDATE chat_messages SET content=<buffer-so-far>,
        status='partial'.

  [Client: chatStore]
    - On meta event → resolve the assistant placeholder row by
      `clientTempId` and bind the server-supplied
      `assistantMessageId` (replace the clientTempId). The user
      placeholder row keeps its `clientTempId`; its server id is
      retrieved on the next conversation read.
    - On each chunk → append delta to the bound assistant row's
      `content` (`status` stays 'pending').
    - On done → bound assistant row.status = 'final'.
    - On error → bound assistant row.status = 'error'; surface
      AIError code via the existing 3-tier visibility (alert +
      __DEV__ console + Sentry TODO marker).
    - On connection lost mid-stream → bound assistant row.status
      = 'partial'; the next reconnect's read pulls the server's
      authoritative 'partial'/'final'/'error' state and reconciles.
    - On replay with the same Idempotency-Key → server returns
      the existing record; no duplicate Gemini call, no double
      charge. (Server skips the plan + quota gate on replay per
      §3 sequence step 2.)
```

### NDJSON wire vocabulary (Drafting checklist 90 + 96, C1 resolution)

`Content-Type: application/x-ndjson`. The HTTP response body is a
single byte stream; each newline-terminated chunk parses as one
JSON object with an `event` discriminator.

| Order | `event` | Fired | Payload |
|---|---|---|---|
| 1 (always first) | `meta` | Right after server-side persistence of the pending assistant + user rows (the persistence steps above), before any LLM token arrives | `{ event: 'meta', assistantMessageId: string, conversationId: string, model: string }` |
| 2 .. N-1 | `chunk` | Per LLM token / partial-text arrival | `{ event: 'chunk', delta: string }` |
| Final (success) | `done` | After the LLM completes and the final UPDATE has committed | `{ event: 'done', inputTokens: number, outputTokens: number, model: string, finishReason: 'stop' \| 'length' \| 'safety' }` |
| Final (failure) | `error` | Mid-stream Gemini / network failure after the meta line | `{ event: 'error', code: AIErrorCode, message: string, recoverable: boolean }` |

**EOF semantics** (NewS1 resolution):

- Stream end = explicit final newline after the `done` or `error`
  line. The server implementation MUST flush a trailing `\n`
  after the final event and before closing the connection.
- The client reads through the newline-terminated `done` / `error`
  line and ignores any further bytes.
- If the client observes a TCP close before a newline-terminated
  `done` / `error` line → treat the local assistant row as
  `partial`; the server's authoritative state is reconciled on
  the next read.
- Parser implementation: split the byte stream on `\n`, dispatch
  on each complete line only. The unterminated trailing remainder
  is NOT flushed on TCP close — see §4.3 — because by the EOF
  contract above its absence already implies transport loss, and
  the right outcome is the `partial` classification.

**Pre-stream errors** (HTTP 401 / 403 / 429): respond with
`Content-Type: application/json` and the existing error shape
`{ error: AIErrorCode, message: string, details?: object }`; the
NDJSON stream never starts in that case. Clients distinguish via
the response's content type.

**Abort**: `AbortController.abort()` closes the fetch; the server
detects the connection close and routes through the 30s safety
timer (the "on client disconnect" final step of the send
sequence). Client UI tags the partial content with "中断しました".

### Offline behavior (I2)

- Read paths: home / coach-tab conversation list / coach-tab message
  history all read from the SQLite mirror. They work offline; the
  copy says "オフライン中" if no network.
- Write paths: the chat send button is disabled when offline; tapping
  surfaces a toast "ネットワーク接続が必要です". `coach-advice` is
  unaffected (lazy on-mount; if offline + no cache, show
  placeholder "オフライン中、 後ほど更新されます").
- The `sync_queue` integration is **NOT** used for chat / advice
  / routine_generations surfaces today. All three are server-
  authoritative (the EF / `supabase-js` writes flow to Supabase
  first; the local mirror is a read cache). The §5.2 Phase 1.0
  design originally listed `routine_generations` as an I2
  exception that would flow through `sync_queue` for offline
  apply, but Phase 1.5 implementation deferred the sync_queue
  resource module + switched to a server-authoritative model
  (online-required apply). See §5.2 Phase 1.5.1 sync note for
  the full rationale + v1.5+ defer plan.

## 4. Design B — LLM client abstraction layer

### 4.1 Interface (Phase 1.1 will refine the signatures)

```typescript
// src/infra/llm/types.ts (new)
export type ChatRole = 'system' | 'user' | 'assistant';
export interface ChatMessage {
  role: ChatRole;
  content: string;

export interface UserContext {
  // §6 fills the shape
  profile: ProfileSnapshot;

  idempotencyKey: string;
  // Phase 1.1: maxOutputTokens, temperature, top_p tunables.
}

export interface AdviceOptions {
  scope: 'weekly' | 'daily';
}

// src/infra/llm/client.ts (new, Phase 1.1)

    messages: ChatMessage[],
    context: UserContext,
    options?: ChatOptions,
  ): AsyncIterable<string>;

  generateStructured<T>(
    prompt: string,
    schema: object, // JSON schema-compatible shape
    context: UserContext,
    options?: ChatOptions,
  ): Promise<T>;

    context: UserContext,
    options: AdviceOptions,
  ): Promise<string>;

// src/infra/llm/geminiFlashClient.ts (new, Phase 1.1)
export class GeminiFlashClient implements LLMClient { ... }
```

### 4.2 Where the implementation lives

The `LLMClient` interface is the **client-facing seam**. The
**actual Gemini calls run server-side in Edge Functions** — the
client never holds the LLM API key. The flow is:

```
React Native client
  → src/infra/llm/geminiFlashClient.ts (LLMClient impl)
  → src/infra/services/aiNutritionService.ts callEdgeFunction
  → Supabase Edge Function (e.g. coach-chat)
  → Gemini API
```

The abstraction's value is on the **client interface**, not on
the network boundary. If a future provider lands (Claude Haiku,
GPT-4o-mini), the server-side EF changes its outbound fetch
target; the client `LLMClient` impl stays the same OR a new
`ClaudeHaikuClient` slots in next to `GeminiFlashClient` and we
flip a feature flag.

### 4.3 Streaming surface (NDJSON, C1 resolution)

`chat()` returns `AsyncIterable<string>` — each `for await` chunk
yielded is the next text delta. Internally the client reads
`Response.body` as a `ReadableStream`, decodes via `TextDecoder`,
splits on `\n`, JSON-parses each line, dispatches on the line's
`event` discriminator:

- `event: 'meta'` — captured, attached to the AsyncIterable's
  side-channel (the caller reads `assistantMessageId` from this
  before any chunk is yielded; the chunks themselves are pure
  text deltas).
- `event: 'chunk'` — yields `delta` to `for await`.
- `event: 'done'` — closes the iterator normally; the side-channel
  carries `inputTokens` / `outputTokens` / `finishReason` for the
  caller's post-stream telemetry.
- `event: 'error'` — closes the iterator with an `AIError` thrown
  on the next `.next()` call.

The implementation also exposes an explicit observer hook for the
`meta` event so the chat store can bind the local pending row to
the server's `assistantMessageId` before chunks start arriving
(this avoids a race where the first `chunk` event lands while the
store is still resolving the row id).

Pre-stream HTTP failures (401 / 403 / 429) reject the AsyncIterable
synchronously on first `.next()` with the existing `AIError`
shape; the NDJSON path is never entered. Mid-stream LLM failures
land via the `error` line above.

The pattern is closer to OpenAI / Anthropic SDK streaming
semantics than raw SSE; the NDJSON choice is documented in §3 and
locked into Phase 1.1's exit criteria.

Parser detail (NewS1 alignment): the implementation splits the
byte stream on `\n` and dispatches one complete line at a time.
The unterminated trailing remainder on TCP close is NOT flushed —
per the §3 EOF contract a final event without a trailing newline
is treated as transport loss and routes to the `partial`
classification, never to a synthetic `done` / `error` reconstruction.

### 4.4 Pattern application

- **Pattern 11** (Error visibility & SDK boundary): the streaming
  error model is documented above with an explicit event
  vocabulary so callers know whether to retry, abort, or
  surface-to-user.
- **Pattern 15.1** (Wrap don't rewrite): the existing five
  Gemini EFs stay in place; they don't get refactored to flow
  through the new `LLMClient`. The new chat / generation /
  advice EFs are net new and use the new abstraction.
- **Pattern 18 facet 11** (preset adoption version compat): the
  `LLMClient` interface lives in this repo, not behind an upstream
  SDK that could pin an incompatible Node version. The
  jest-expo / jest-30 incompatibility we hit in v1.4 Stage 5.2 is
  the precedent.
- **Pattern 25** (Type-system widening): keep `UserContext` as a
  structural type so a future caller can omit fields when the
  context build path is optional.
- **Drafting checklist 93**: provider-switchable abstraction is
  in from the first commit, not bolted on later.

### 4.5 Migration option for existing EFs

The five existing Gemini EFs (`estimate-nutrition`,
`estimate-nutrition-vision`, `nutrition-advice`,
`generate-workout-menu`, `generate-weekly-report`) **stay on
their current bare-`fetch` pattern through v1.5**. A v1.5+
candidate (tracked in `docs/plans/v1.5_roadmap.md`) is to
gradually back-port them onto the `LLMClient` — but only after
the new interface has shipped, been dogfooded for at least one
release, and exposed any gaps.

Indexes:
- `chat_messages (conversation_id, created_at)` for chronological history reads.
- `chat_messages (idempotency_key) where idempotency_key is not null`
  — partial unique index for the C2 idempotency lookup. The 24h
  retention window is enforced by a background cleanup job that
  runs `UPDATE chat_messages SET idempotency_key = NULL WHERE
  idempotency_key IS NOT NULL AND created_at < now() - interval
  '24 hours'`. The partial index predicate `where idempotency_key
  is not null` causes NULLed rows to drop out of the index
  automatically — the row itself is preserved (the message history
  stays intact); the key becomes reusable for a new request. This
  is the implementable mechanism for "delete from the partial
  index" since PostgreSQL does not expose direct partial-index
  entry deletion.
- `coach_advice (user_id, scope, period_start desc)` — paired with the unique constraint above; serves the "latest for this user / scope / period" read in one index seek.
- `routine_generations (user_id, status, created_at desc)`.
- `routine_generations (user_id, idempotency_key) where idempotency_key is not null`
  — partial unique index added in Phase 1.5 migration
  `20260519000000`. Same retention contract as the chat_messages
  partial unique: the hourly `cleanup-idempotency-keys` EF NULLs
  rows older than 24 h on BOTH tables, and the partial predicate
  drops NULLed rows from each keyspace automatically. Drafting
  109 + Phase 1.5.1 sync note in §10 documents the post-hoc
  migration rationale.

### 5.1.1 Persisted snapshot scope (I5 + Drafting checklist 95)

`coach_advice.source_data_snapshot_json` and
`routine_generations.prompt_context_json` are the only persisted
records of "what we sent to Gemini." They **mirror the projected
safe subset**, not the client-side raw context. The same
projection that §6.4 applies on the wire applies here.

Allowed in the persisted snapshot:

- `profile.ageRange` (5-year bucket)
- `profile.sex`, `profile.heightCm`, `profile.weightKg`,
  `profile.goalType`, `profile.activityLevel`,
  `profile.trainingDaysPerWeek`
- `targets.{calories, proteinG, fatG, carbG}`
- `recentMeals.last7DaysAverage`,
  `recentMeals.todaySoFar`,

- `recentWorkouts.{last14DaysSessions, routineNames,
  weeklyVolumeKgRepsByMuscle?}`

- `profile.display_name`, `profile.supabase_uid`, exact birth date,
  email, device identifiers.
- Raw food names beyond the top-5 frequent.
- Any unstructured user-typed text that hasn't passed the
  projection.

snapshot is then written from the same projected value. There is
no path where the raw context lands in durable storage.

## 6. Design D — Context injection

### 6.1 Token budget

Gemini 2.5 Flash has a 1M-token input window in theory, but
**latency + cost scale with context**. We hold the **prompt + context
budget ≤ 6,000 tokens** to keep p99 streaming first-chunk under
2 seconds.

### 6.2 Context shape

```typescript
interface UserContext {
  profile: {
    ageRange: '10-14' | '15-19' | '20-24' | ...; // PII minimization
    sex: 'male' | 'female' | 'other';
    heightCm: number;
    weightKg: number;
    goalType: 'cut' | 'bulk' | 'maintain' | 'recomp';
    activityLevel: 'sedentary' | 'light' | 'moderate' | 'active' | 'very_active';
    trainingDaysPerWeek: number;
  };
  targets: {
    calories: number;
    proteinG: number;
    fatG: number;
    carbG: number;
  };
  recentMeals: {
    last7DaysAverage: { calories: number; proteinG: number; fatG: number; carbG: number };
    todaySoFar?: { calories: number; proteinG: number; fatG: number; carbG: number };
    topFrequentNames: string[]; // top 5 by frequency
  };
  recentWorkouts: {
    last14DaysSessions: number;
    routineNames: string[]; // unique last-14d
    weeklyVolumeKgRepsByMuscle?: Record<string, number>; // optional, Plus+
  };
  recentWeightTrend: {
    last14DaysKgChange: number; // (weight today) - (weight 14d ago)
  };
}
```

### 6.3 Builder

`src/infra/llm/contextBuilder.ts` (new, Phase 1.1):

```typescript
export async function buildUserContext(
  profileId: string,
  options?: { mealDays?: number; workoutDays?: number },
): Promise<UserContext> { ... }
```

Implementation reads from the existing repositories:
`profileRepository`, `nutritionRepository`, `workoutRepository`,
`bodyLogRepository`. No new SQL — composes existing helpers.

### 6.4 PII minimization at Gemini boundary

What we send:
- Age range (5-year bucket), not birth year.
- Goal, activity level, training frequency.
- Numeric weight (kg) and height (cm). These are needed for
  meaningful coaching; cannot be redacted.
- Nutrient + workout numeric summaries.

What we don't send:
- `profiles.display_name`, `supabase_uid`, exact birth date,
  email address, device identifiers.
- Raw food names beyond the top-5 frequent list — that prevents
  inference of restaurant location / lifestyle from individual
  rare meals.

The Edge Function does the redaction (the client builds the full
context, the EF projects to the safe subset before assembling the
Gemini prompt).

**Persisted snapshot boundary (I5 resolution)**: the persisted
snapshot in `coach_advice.source_data_snapshot_json` and
`routine_generations.prompt_context_json` mirrors **the projected
safe subset, not the client-side raw context**. See §5.1.1 for the
explicit allowed / forbidden field lists. There is no path where
the raw context lands in durable storage — the EF projection
runs before both the Gemini call and the snapshot write, and the
projected value is the single value that flows into both.

2. Server-side の `public_foods` table は user-submitted のみで、
   八訂 seed は server に存在しない。
3. Cross-EF HTTP は I6 fix で禁止済 (double-charge / TOCTOU /
   Pattern 15.1 violation)。

Stage 1 の architectural answer: **DB lookup 機能は Stage 2 epic に
完全委譲**、 Stage 1 EF は Gemini の公知メニュー知識 + 注釈 tag で
運用する。

   Supabase に追加。

  kickoff. `DECISION NEEDED` markers exist for things the doc
  doesn't pretend to know; they cannot coexist with prose that
  pretends to know.

- **97 — Quota-charged streaming operations coordinate
  accounting + persistence timing**: when a server-side
  operation charges a quota AND emits a stream, the
  "optimistic client append + persist on finish" pattern is a
  usage-charged silent-data-loss class. Quota is charged when
  generation starts; persistence has to be authoritative on
  the server before the first chunk goes out so that a mid-
  stream client kill leaves a recoverable trace, not a
  ghost-charge. The fix landed in this revision: persist user
  message + pending assistant row BEFORE the LLM call; emit
  the assistantMessageId via the `meta` event; reconcile
  partial / final / error states server-side.

- **98 — Idempotency replay must precede mutable state
  changes**: retry / replay lookup must run immediately after
  authentication, before plan gate and before quota gate. If
  the gates run first, a legitimate retry after the original
  request consumed quota (or after the user's plan state
  changed) gets rejected with `quota_exceeded` / `plan_required`
  even though the server already holds an authoritative result.
  That breaks the "no double charge / replay existing record"
  contract. Phase 1.0.1 → 1.0.2 fix: reorder §3 server step 2
  (idempotency) above steps 3-4 (plan + quota); replay path
  short-circuits with NO ai_usage_logs INSERT, NO Gemini call,
  NO gate evaluation.

- **99 — Cross-runtime resource availability must be verified
  before claiming code re-use**: when a doc says "re-use helper
  X across runtimes", the helper's full dependency graph must
  be re-checked in BOTH target runtimes before the claim lands.
  The Phase 1.0.1 I6 fix wrote "re-use `searchFoods` /
  `findByExactName` as a Deno-side helper, not via cross-EF
  HTTP" without tracing `foodRepository.ts → getDatabase() →
  expo-sqlite`, which is RN/Expo-only and cannot run inside a
  Deno EF. Phase 1.0.2 fix: §8.2 retracted the in-EF lookup
  and assigned the food-DB hook responsibility to Stage 2.
  Lesson generalizes to any "shared helper between client +
  EF" claim — Hermes / RN runtime requirements (expo-*,
  react-native-*, native-module bindings) and Deno EF
  requirements (no node_modules, no native modules, fetch/web
  primitives only) are not interchangeable, and a helper that
  imports from one side cannot serve the other without an
  explicit runtime-neutral refactor.

- **100 — Race-safe write ordering for idempotent operations**:
  when an EF persists multiple rows as part of one
  user-facing intent AND one of those rows carries the
  idempotency key (via a partial unique index), insert THAT row
  FIRST so the unique constraint fires before any other mutable
  side effect lands. Concurrent duplicate requests with the same
  key then bounce at the constraint check — second-flight INSERTs,
  external API calls, ai_usage_logs inserts, etc. never run for
  the loser. Phase 1.1 Codex round 1 C2 fix is the precedent:
  the `coach-chat` EF originally INSERTed the user row before
  the assistant placeholder; same-key concurrent retries then
  duplicated the user row even though the assistant constraint
  caught them. Swapping to assistant-first + user-second made
  the race rejection happen before the user row was touched.

- **101 — PII projection needs per-field type guards + cardinality
  bounds, not just per-key allowlists**: a "permitted-keys" set
  is necessary but not sufficient defense. A malicious or
  compromised client can stuff a nested object, an unbounded
  string, or a thousand-entry array under an "allowed" scalar
  key (`profile.ageRange`, `targets.calories`,
  `recentWorkouts.routineNames`, etc.) and pass through. Each
  permitted field needs (a) an explicit type guard
  (`asFiniteNumber`, `asEnum`, `asBoundedString`, structured
  projector for nested shapes) and (b) a cardinality / length
  bound for arrays + records. Phase 1.1 Codex round 2 Critical
  fix + Codex round 3 Important #1 fix are the precedent: the
  v1 server-side projection passed values verbatim under the
  allowed-key gate; the hardening rebuilt every field as an
  explicit coercion (returning a safe default on mismatch) and
  added bounds like `ROUTINE_NAMES_MAX = 10` /
  `WEEKLY_VOLUME_MAX_KEYS = 20` /
  `TOP_FREQUENT_NAMES_MAX = 5`.

- **102 — Supabase JS returns `{ data, error }`; `try/catch` alone
  is insufficient**: `@supabase/supabase-js` does NOT throw on a
  PostgREST failure (RLS reject, unique violation, check
  constraint, etc.). It returns `{ data: null, error:
  PostgrestError }`. A naive `try { await client.from(...) }
  catch {}` block silently drops every Postgres-level error and
  the downstream code reads `data` as null. EVERY DB write must
  check `error` explicitly + emit a diagnostic
  (`console.warn` for Supabase log capture, optional Sentry
  breadcrumb later) + decide a caller-side fallback (500 to
  client, skip-and-continue, rollback). Phase 1.1 Codex round 2
  Important #2 fix is the precedent: STEP 6 user INSERT + STEP 7
  ai_usage_logs INSERT + the streaming UPDATEs all gained
  `{ error }` checks; without them the EF could stream a
  successful response while quota was undercounted or the
  assistant row stayed stuck in `pending`.

- **103 — Compensation writes need the same defensive depth as
  main writes**: when the main success path is `{ error }`-
  checked but rollback / cleanup writes still rely on `try /
  catch` only, a recovery write failure silently leaks state
  (a stale `pending` row stays in the index for 24 h, a new
  conversation lingers as an orphan, etc.). Extract every
  compensation write into a dedicated helper that performs the
  same `{ error }` check + the same `console.warn` diagnostic,
  with a best-effort suffix so the helper never throws back to
  its caller (the main path already owns the user-facing
  failure response). Phase 1.1 Codex round 2 Important #1 fix +
  Codex round 3 Important #2/#3 fix + Codex round 4 Important
  #2 fix are the precedent: the three helpers
  `failAssistantRowBestEffort`, `deleteConversationBestEffort`,
  `deleteUserMessageByIdBestEffort` consolidate the cleanup
  paths and make their failure modes visible in the Supabase
  logs; the last helper deletes by primary key (the row id
  captured at insert time) rather than by content match, so
  the rollback can't collateral-damage older identical user
  messages in the same conversation.

- **104 — 環境依存 state は server-authoritative、 client は read-only**:
  時刻 / TZ / locale / quota / race-safe ordering 等、 device 差や
  concurrent 競合の影響を受ける state は server-side で compute / decide
  し、 client は read-only にする。 Client-side で同 logic を実装すると、
  device clock skew + locale 差 + DST timing + concurrent request で
  state mismatch が起き、 quota / data integrity / UX consistency に
  影響する。 必要に応じて client-side mirror (jest test 用) を持つ
  場合は、 production routing には影響させない。
  Phase 1.2 で client-side が UX-natural user-first ordering を採用し
  Drafting 100 (server race-safe ordering) を mirror しなかった例 +
  Phase 1.4 で coach-advice EF が period_start を server-side で
  compute + client-side daily short-circuit を撤去した例 (Codex round
  1 Important #2 fix) が precedent。

- **105 — Sensitive helper は supabase/functions/_shared/ で集約**:
  PII projection / quota gating / idempotency lookup / tz computation
  等の sensitive helper は単一 EF に inline せず、
  supabase/functions/_shared/ に抽出して 全 EF が import で re-use
  する。 これにより per-field type guards (Drafting 101) や error
  check (Drafting 102) の single source of truth が保たれ、 helper
  改善が全 EF に自動で伝播する。
  Phase 1.4 で coach-chat の projectContextSafeSubset を
  _shared/projectContext.ts に抽出 + computePeriodStart を
  _shared/tzPeriod.ts に抽出し、 coach-advice が同 helper を re-use
  した pattern が precedent。

- **106 — Client-side cache は必ず userId-scoped + logout で reset**:
  Zustand / SQLite / MMKV 等の client-side cache で user-specific
  data を保持する場合、 cache key に必ず userId prefix を含める。
  加えて、 logout flow で当該 cache の clear (reset action) を呼ぶ
  こと。 これにより、 同 device で 別 user に切り替わった際に前 user
  の sensitive data が leak しない (cross-account contamination 防止)。
  Phase 1.4 で coachAdviceStore が ${userId}:${scope}:${period} 化 +
  authStore.logout から lazy-import で reset() を呼ぶ pattern (Codex
  round 1 Critical fix) が precedent。

- **107 — Doc spec と implementation reality の gap は explicit な
  reality note で trace**: Doc を「intent + future state」 として
  書く場合、 implementation が現状追従していない field / type /
  behavior は 「Phase X.Y で実装予定」 等の reality note で明示し、
  SSoT 違反 (Pattern 18 facet 12) を起こさない。
  Phase 1.4.1 で §9.1 numeric advice keys が spec 追加されたが
  subscriptionService.ts は boolean のみのため reality note を
  追加した例 + Phase 1.5 で §5.2 sync_queue が routine_generations
  の intent として書かれていたが resource module 未実装のため
  server-authoritative model に切替 + v1.5+ candidate に defer plan
  を documenting した例が precedent。

- **108 — 新 EF が既存 EF の pattern を re-use する場合、 既存 EF
  にも同 bug が無いか cross-audit する**: 新 EF 設計時に「同 pattern
  を使う既存 EF が同 bug を持っていないか」 を必ず recon に含める。
  Phase 1.5 で coach-routine 実装中に `.eq('user_id', userId)` bug
  が判明、 既存 coach-chat (Phase 1.1) + coach-advice (Phase 1.4) に
  も同 bug があり同時 fix した precedent。 これら既存 2 EF は累計
  5 round の Codex review で見落とされていた critical bug が Phase
  1.5 で初めて検出された。

- **109 — EF が idempotency replay を採用する table は、
  idempotency_key column + partial unique index を initial migration
  で必ず含める**: 新 table を作る migration で、 EF が idempotency
  replay を pattern として使う場合、 idempotency_key column + partial
  unique index は initial migration に必ず含める。 後追い migration
  で追加するのは migration sequencing が複雑化し、 retention cleanup
  (cleanup-idempotency-keys EF) の更新も必要になる。
  Phase 1.5 で routine_generations 用に追加 migration 20260519000000
  を作った + cleanup-idempotency-keys EF を両 table NULL 対応に拡張
  した precedent (Phase 1.1 の 5 tables 統合 migration では
  idempotency_key column を routine_generations に入れ忘れていた)。
